**Severity:** Critical

## Evidence

- The user-visible error strings map to [`src/components/subscription/paywall-gate.tsx`](/Users/phaedrus/.codex/worktrees/50f2/volume/src/components/subscription/paywall-gate.tsx), specifically the `convex_auth` bootstrap timeout branch.
- The production `sign-in` HTML includes the live Clerk publishable key but no obvious production Convex URL, while the fetched client chunk still contains:
  - `getToken({ template: "convex" })`
  - `d("NEXT_PUBLIC_CONVEX_URL")`
  - fallback `https://local-build.convex.cloud`
  - hosted check via `d("VERCEL_ENV")`
- In browser code, `VERCEL_ENV` is not guaranteed to be available unless explicitly exposed. That makes the hosted check unreliable in the client runtime.
- Sentry project `volume` has no matching issue for `Subscription gate bootstrap timed out` or `PaywallGate` in the last 14 to 30 days.
- The closest auth issue is `VOLUME-2` (`Unauthorized: No valid authentication token. Please sign in again.`), but its latest event is tagged `environment=dev`, not production.

## Root Cause

The previous client public-service helper used `VERCEL_ENV` to decide whether a client runtime was a hosted deployment. In the browser, that signal is unreliable because it is not a public env var. If `NEXT_PUBLIC_CONVEX_URL` is missing from the client runtime, the helper can silently fall back to `https://local-build.convex.cloud` instead of failing closed.

That creates a split-brain auth state:

1. Clerk server auth still treats the user as signed in.
2. `PaywallGate` renders protected routes.
3. Convex client auth never becomes ready against the wrong endpoint.
4. The gate times out in the `convex_auth` phase and shows the recovery UI the user reported.

## Why Sentry Did Not Catch This Cleanly

- The timeout UI should call `reportError(...)`, but the `volume` Sentry project shows no corresponding production issue.
- Current evidence points to production browser Sentry not being active or not initializing for this path, rather than the timeout branch never running.
- This incident therefore exposed two separate gaps:
  - production auth bootstrap could silently target a local-build Convex endpoint
  - production client Sentry is not giving reliable visibility for this user-facing failure

## Fix

- Split public runtime config into [`src/lib/public-service-config.client.ts`](/Users/phaedrus/.codex/worktrees/50f2/volume/src/lib/public-service-config.client.ts) and [`src/lib/public-service-config.server.ts`](/Users/phaedrus/.codex/worktrees/50f2/volume/src/lib/public-service-config.server.ts) so browser code cannot accidentally rely on server-only env semantics.
- Updated the client helper to detect hosted browser runtimes from `window.location.hostname` instead of relying on `VERCEL_ENV`.
- Kept the server/build-time `VERCEL_ENV` guard as a secondary signal.
- Added regression coverage in [`src/lib/public-service-config.test.ts`](/Users/phaedrus/.codex/worktrees/50f2/volume/src/lib/public-service-config.test.ts) for hosted browser hostnames missing public Clerk/Convex config.
