# Coach Production Incident

## Summary

Coach turns were failing before the planner could persist any response, so the
Coach agent appeared to hang or return a generic error instead of responding.
The immediate cause was malformed planner history: the request window could
start with an orphaned `tool_result` block after truncation, and Anthropic
rejected that transcript with HTTP 400. The observability gap was separate:
server-side Canary capture was never initialized through `reportError()`, and
Next instrumentation only initialized Canary when `CANARY_*` env vars existed,
even though production only had `NEXT_PUBLIC_CANARY_*`.

## Timeline (UTC)

- 2026-03-27 22:02, 22:07, 22:33: production coach requests stopped after
  `coachSessions:getSessionMessages*`; no subsequent tool work or
  `coachSessions:addMessage` was recorded.
- 2026-03-27 23:xx: `/api/health` returned 503 with `coachRuntime=pass` and
  `sentry=fail`, showing the runtime was up but server error-tracking health was
  misconfigured.
- 2026-03-28 00:xx: live session history was replayed locally against the
  production planner and reproduced the provider failure:
  `Each tool_result block must have a corresponding tool_use block`.
- 2026-03-28 00:20: fix prepared with tests for whole-turn history slicing,
  server-side Canary init, handled planner failure reporting, and health checks.

## Evidence

- Production Convex logs showed failures ended after session-history reads and
  before any planner side effects or message persistence.
- Replay of the active production coach session reproduced an OpenRouter /
  Anthropic 400 on the planner call with an orphaned `tool_result`.
- `buildCoachHistory()` used `slice(-20)` on raw unsummarized messages, which
  can cut through the middle of a tool-using turn.
- Vercel production env contained `NEXT_PUBLIC_CANARY_ENDPOINT` and
  `NEXT_PUBLIC_CANARY_API_KEY`, but not `CANARY_ENDPOINT` or `CANARY_API_KEY`.
- `src/lib/analytics.ts` only initialized Canary in the browser path, so
  server-side `reportError()` would call `captureException()` without an
  initialized Canary client.
- `src/instrumentation.ts` only initialized Canary from `CANARY_*`, so
  unhandled request errors also missed Canary in production.
- The fatal planner failure was handled and returned as a 500 stream, so
  `onRequestError` would not have captured it even if instrumentation had been
  configured correctly.

## Hypotheses Considered

1. OpenRouter API key or runtime outage. Rejected: health check showed
   `coachRuntime=pass`.
2. Convex persistence failure. Rejected: failing requests stopped before any
   persistence mutation and reproduced locally during the planner call.
3. Malformed planner transcript after history truncation. Confirmed by replay
   and by the exact provider error referencing orphaned `tool_result` blocks.
4. Canary ingestion outage. Rejected: the gap was local initialization and env
   resolution, not the Canary endpoint itself.

## Root Cause

`buildCoachHistory()` truncated unsummarized messages by raw message count
instead of whole turns. When the cutoff landed in the middle of a prior
tool-using assistant turn, the planner request began with a `tool_result`
without the preceding `tool_use`. Anthropic rejected the transcript before any
tool execution or response persistence.

## Why Canary Did Not Auto-Triage

1. Production only configured `NEXT_PUBLIC_CANARY_*`.
2. `src/instrumentation.ts` looked only for `CANARY_*`, so unhandled request
   capture never initialized Canary on the server.
3. `src/lib/analytics.ts` initialized Canary only in the browser path, so
   server-side `reportError()` also no-op'd for Canary capture.
4. The planner failure path was handled and returned as a 500 response, so
   `onRequestError` would not have seen it anyway.
5. `/api/health` and `scripts/verify-env.sh` still treated Sentry-only server
   DSNs as the runtime error-tracking gate, which masked the actual Canary
   wiring the app was attempting to use.

## Fix

- Added a whole-turn history slicer so planner context never starts mid-turn.
- Aligned conversation summarization to the same whole-turn boundary.
- Added shared Canary init-option resolution with server fallback to
  `NEXT_PUBLIC_CANARY_*` when dedicated server vars are absent.
- Updated both `reportError()` and Next instrumentation to initialize Canary on
  the server.
- Added explicit `reportError()` capture for handled planner failures with no
  tool execution.
- Updated health and env verification to check runtime error tracking instead of
  Sentry-only server DSNs.
- Added resolver aliases so the Canary workspace package is visible to Vitest
  and TypeScript.
