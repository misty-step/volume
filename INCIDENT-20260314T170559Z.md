**Date:** 2026-03-14
**Severity:** Critical
**Status:** Root cause identified, code fix prepared, production config still needs verification

## Evidence

- The user-visible error strings map to [`src/components/subscription/paywall-gate.tsx`](src/components/subscription/paywall-gate.tsx), specifically the `convex_auth` bootstrap timeout branch.
- The production client chunk still contained:
  - `getToken({ template: "convex" })`
  - dynamic env lookup for `NEXT_PUBLIC_CONVEX_URL`
  - fallback `https://local-build.convex.cloud`
- PR E2E trace artifacts showed the same pattern in CI:
  - Clerk booted with `comic-terrapin-21.clerk.accounts.dev`
  - Convex logged `[CONVEX FATAL ERROR] Couldn't parse deployment name local-build`
  - the app rendered the same `PaywallGate` bootstrap-timeout recovery screen seen in production
- In Next.js/Turbopack app code, public env values must be read via literal access like `process.env.NEXT_PUBLIC_CONVEX_URL`. Computed access like `process.env[key]` is opaque to the compiler and can silently drop the value from the bundle.
- Sentry project `volume` has no matching issue for `Subscription gate bootstrap timed out` or `PaywallGate` in the last 14 to 30 days.
- The closest auth issue is `VOLUME-2` (`Unauthorized: No valid authentication token. Please sign in again.`), but its latest event is tagged `environment=dev`, not production.

## Root Cause

The public runtime config helpers were using computed env reads like `process.env[key]`. In Next.js app code, that breaks static env exposure for browser-facing values. As a result, `NEXT_PUBLIC_CONVEX_URL` and `NEXT_PUBLIC_CLERK_PUBLISHABLE_KEY` could be present in deployment or CI process env while still disappearing from the compiled app bundle.

That left the runtime on fallback values:

- Clerk could boot against the local-build frontend API fallback
- Convex could boot against `https://local-build.convex.cloud`
- hosted/runtime detection had less signal than intended once the public vars were already missing

That creates a split-brain auth state:

1. Clerk server auth still treats the user as signed in.
2. `PaywallGate` renders protected routes.
3. Convex client auth never becomes ready against the wrong endpoint.
4. The gate times out in the `convex_auth` phase and shows the recovery UI the user reported.

## Why Sentry Did Not Catch This Cleanly

- The timeout UI should call `reportError(...)`, but the `volume` Sentry project shows no corresponding production issue.
- Current evidence points to production browser Sentry not being active or not initializing for this path, rather than the timeout branch never running.
- This incident therefore exposed two separate gaps:
  - production auth bootstrap could silently target a local-build Convex endpoint
  - production client Sentry is not giving reliable visibility for this user-facing failure

## Fix

- Split public runtime config into [`src/lib/public-service-config.client.ts`](src/lib/public-service-config.client.ts) and [`src/lib/public-service-config.server.ts`](src/lib/public-service-config.server.ts) so browser and server helpers stay narrow.
- Replaced computed `process.env[key]` lookups with literal `process.env.NEXT_PUBLIC_*` access in the public runtime helpers and Sentry release resolution.
- Kept the hostname-based hosted-runtime guard for browser fallback behavior and the server-side `VERCEL_ENV` guard for hosted builds.
- Added an ESLint rule plus regression tests to block computed or aliased `process.env` access in `src/`, so Next app code must use direct `process.env.NAME` reads.
- Kept targeted unit coverage in [`src/lib/public-service-config.client.test.ts`](src/lib/public-service-config.client.test.ts), [`src/lib/public-service-config.server.test.ts`](src/lib/public-service-config.server.test.ts), [`src/lib/sentry.test.ts`](src/lib/sentry.test.ts), and [`eslint.config.test.ts`](eslint.config.test.ts).
